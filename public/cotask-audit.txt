# 1. Simple Product Explanation

CoTask lets a client be somewhere without physically being there by hiring a trusted Local Agent to complete a real-world task remotely through messages, live video, proof, reviews, and secure payment.

*   **Client:** The user who needs something done remotely (e.g., viewing an apartment, inspecting a car, running an errand). They pay for the service.
*   **Local Agent:** The verified user on the ground who performs the task, streams video if required, and submits proof of completion to earn money.
*   **Task:** The specific job to be done. It can be an Open Task (broadcast to many) or a Direct Hire (sent to one).
*   **Direct Hire:** A private request sent directly from a Client to a specific Local Agent they found on the Discover page.
*   **Open Task:** A public job board post created by a Client. Multiple Local Agents can see it, submit proposals (with rates/times), and the Client chooses one winner.
*   **My Tasks:** The central dashboard where both Clients and Local Agents track their ongoing, upcoming, and past tasks (aggregating both Direct Hires and Open Tasks).
*   **Secure Payment:** The financial system (via Stripe) where a Client's money is captured and held in escrow *before* work begins, and released to the Local Agent only after the task is approved.
*   **Live Session:** A scheduled or ad-hoc video call (using Daily.co) where the Local Agent streams their perspective to the Client in real-time.
*   **Proof:** Photos, videos, or notes uploaded by the Local Agent to prove the task was completed successfully, triggering the Client's approval phase.
*   **Review:** A 1-5 star rating and text feedback left by either party after the task is completed, affecting public profiles.

# 6. Current Booking / Task Creation Flow

**Direct Hire (`CreateBooking.jsx`)**
*   **Data collected:** Category, date, time, duration, stream mode (live/no camera), transport needs, equipment, notes.
*   **Required:** Avatar Profile ID, Category.
*   **Errors/Missing Data:** Blocked by frontend guard (fixed recently). If missing, shows empty state.
*   **Payment Trigger:** Currently triggered *immediately* via `createAndPay`.
*   **Success State:** Redirects to Stripe Checkout (or shows generic success if free test).

**Open Task (`PostJob.jsx`)**
*   **Data collected:** Title, description, category, location, budget range, skills/equipment, scheduled date.
*   **Required:** Title, description, category.
*   **Errors/Missing Data:** Form validation prevents submission.
*   **Payment Trigger:** None at creation. Triggered only when a proposal is accepted.
*   **Success State:** Redirects to the published `JobDetail.js` page.

# 7. Current Payment Placement

*   **Where does payment currently appear?**
    *   Direct Hire: At the very end of the `CreateBooking` form.
    *   Open Task: Inside `JobDetail` when clicking "Hire" on a proposal.
*   **Is payment currently paused?** Attempted but paused. The UI has buttons, but test-mode flows were interrupted by the `avatar_profile_id` bug.
*   **Terminology:**
    *   *Secure Payment Required:* Client needs to pay.
    *   *Secure Payment Held:* Money is captured by Stripe and sitting in Base44's connected platform account.
    *   *Released/Paid:* Task approved, money sent to Agent's Base44 balance (and eventually paid out via Stripe Connect).
    *   *Disputed:* Task paused, funds frozen pending Admin review.
*   **Recommendation for Production V1:** **Model B (Client sends request first, Local Agent accepts, then client funds Secure Payment).**
    *   *Why?* Paying upfront for a Direct Hire request that an agent might decline (because they are busy, asleep, or don't want the job) creates massive friction, requires complex automatic voiding/refunding logic, and ties up user funds unnecessarily.
    *   *Flow:* Client Requests -> Status: `pending`. Agent Accepts -> Status: `accepted_awaiting_payment`. Client Pays -> Status: `scheduled`/`in_progress`.

# 8. Current Task Status Lifecycle

**Direct Hire (`Booking` entity)**
1.  `pending` (Request sent, waiting for Agent)
2.  `accepted` (Agent agreed, waiting for payment) -> *Ideal state, currently bypassed by forced upfront payment*
3.  `scheduled` (Paid, waiting for date/time)
4.  `in_progress` (Agent is working/traveling)
5.  `live` (Video stream active)
6.  `awaiting_approval` (Proof submitted by Agent)
7.  `completed` (Client approved, funds released)
8.  `disputed` (Client raised issue)
9.  `cancelled` / `declined` (Dead ends)

**Open Task (`JobPost` entity)**
1.  `open` (Receiving proposals)
2.  `awaiting_approval` (Winner selected, awaiting client payment)
3.  `in_progress` (Paid, work started)
4.  `completed` (Approved)
5.  `disputed`
6.  `cancelled`

*Note: The terminology and statuses between the two schemas are slightly mismatched (e.g., `JobPost` lacks a specific `scheduled` state, `Booking` lacks an `open` state). Unifying them is highly recommended.*

# 11. Current Data Flow Map

**Direct Hire:**
`CreateBooking` (Frontend) -> `createBooking` (Backend) -> Creates `Booking` Record -> Stripe Checkout -> Webhook marks `payment_status: held` -> Agent sees in `AvatarRequests` -> Agent Clicks Accept -> Messaging opens -> `JobActionCard` handles lifecycle updates -> Proof uploaded -> `releaseTaskPayment` (Backend) updates `AvatarProfile.total_earnings` and marks `Booking` completed.

**Open Task:**
`PostJob` (Frontend) -> Creates `JobPost` Record -> Agent sees in `JobMarketplace` -> Agent calls `applyToJob` -> Creates `JobApplication` Record -> Client sees apps in `JobDetail` -> Client accepts -> Stripe Checkout -> Webhook marks `JobPost.escrow_status: captured` -> `JobActionCard` handles lifecycle -> `releaseTaskPayment` (Backend).

*Weak Points:* Two completely separate data structures for the exact same lifecycle. If you want to add a feature (e.g., "Add 30 minutes to task time"), you have to build it twice (once for `Booking`, once for `JobPost`).

# 13. Recommended Clean Production V1 Process

To make CoTask feel like a premium, reliable product, the flow must be standardized.

**The Golden Rule of V1:** No money moves until both parties agree to the scope.

## 1. Client Direct Hire (The Fix)
1.  Client clicks "Request Direct Hire" on a profile.
2.  Client fills out details (Date, Time, Scope).
3.  Client clicks **"Send Request"** (NO PAYMENT YET).
4.  Local Agent receives notification: "New Request from [Name]".
5.  Agent reviews details and clicks **"Accept & Request Payment"**.
6.  Client receives notification: "Agent Accepted! Fund the escrow to confirm."
7.  Client clicks **"Fund Secure Payment"** -> Stripe Checkout.
8.  Task becomes `scheduled` / `in_progress`.

## 2. Client Open Task
1.  Client posts Open Task (NO PAYMENT YET).
2.  Local Agents submit proposals (Rate + Message).
3.  Client reviews proposals and selects a winner.
4.  Client clicks **"Hire & Fund Escrow"** -> Stripe Checkout.
5.  Task becomes `scheduled` / `in_progress`.

## 3. Execution & Live Session
1.  At scheduled time, both parties see a **"Join Live Session"** button in their Task Detail page.
2.  Button opens `LiveStreamStudio` (Agent broadcasts) / `ClientLiveView` (Client watches).

## 4. Completion & Approval
1.  Agent clicks **"Submit Proof of Completion"** (Uploads photo/notes).
2.  Client gets notification. Clicks **"Approve & Release Funds"**.
3.  Base44 backend transfers held Stripe funds to Agent's connected account balance.
4.  Both prompted to leave a Review.

# 16. Final Summary for ChatGPT

**Current State:** CoTask's core architecture is built, but the UX is currently messy because the app maintains two completely separate workflows and database entities for the same concept (`Booking` for Direct Hires, `JobPost` for Open Tasks). Navigation is cluttered with redundant roles (User vs Avatar vs Enterprise), and payment logic is awkwardly placed.

**What is Broken/Risky:** The biggest flaw right now is that Direct Hire forces the Client to pay via Stripe *before* the Local Agent even accepts the request. If the agent declines, complex refunds are required.

**How it Should Work:** The Golden Rule for V1 should be implemented: **No money moves until both parties agree.** Direct Hires should send a free "Request" to the agent. Once the agent accepts, the Client is prompted to "Fund Secure Payment". Open Tasks should follow the same logic (Post -> Agent Proposes -> Client Accepts & Funds).

**Next Fix:** We must execute Phase B: Decouple Stripe checkout from the `CreateBooking` form submission, and move the "Fund Secure Payment" button to the Task Detail page, revealing it only after the Agent's status becomes `accepted`. Do not build new features or redesign UI until this core payment timing is resolved.
